## 一、审查结论

上一轮（84/100）修复的是**正确性/一致性**层面的零散问题（魔法数字、孤儿键、文案、色盲标签、响应式 600px 断点）。
用户本轮「整体不满意」指向的是**信息架构与视觉层级**——即页面**怎么组织、怎么被看懂**，而非单点 bug。

核心矛盾：**单页长滚动 + 结果区无分区 + 设计 token 仅限颜色**，导致页面"能跑但难读、能看但平庸"。

| 维度 | 现状评分 | 关键问题 |
|------|----------|----------|
| 信息架构 | 中下 | 单页垂直堆叠（URL→进度→日志→结果大块），无分区/分页；结果一旦生成页面极长 |
| 视觉层级 | 中 | 仅颜色 token，无 spacing/typography/radius token；卡片靠边框区分，缺阴影与层次 |
| 布局合理性 | 中下 | 结果区 `col_img(1):col_text(2)`，质检大表格挤在 1/3 列宽 expander 内必溢出；稿件 `height=500` 容器与统计割裂 |
| 交互反馈 | 中 | 运行中 `@st.fragment(run_every=3)` 全局重绘导致日志区每 3s 闪烁；首启 45-90s 仅一行 spinner |
| 可维护性（影响"改不动") | 低 | 1967 行单文件，UI 与流水线/线程/CSS 混杂，`render_*` 强耦合 session_state |

## 二、深层问题清单（按影响面）

### A. 信息架构 / 布局
1. **单页长滚动无分区**：`render_main`(URL) → `render_progress`(灯+条) → `render_logs` → `render_results`(质检仪表盘+稿件+配图+文件清单) 垂直堆叠。生成结果后页面极长，用户需反复滚动在"监控"与"成果"间切换。
2. **结果区列宽不合理**：`render_results` 用 `col_img(1):col_text(2)`。质检仪表盘的 8 列 Markdown 表格被塞进 `col_img`（1/3 宽）的 `st.expander`，列必横向溢出/折行，几乎不可读。
3. **运行态与完成态共用入口**：`main()` 情况2/情况3 分支渲染，但「下一步」按钮嵌在 `run_every=3` fragment 内，每 3s 重跑 fragment → 日志区频繁整体重绘闪烁。
4. **侧栏与主流割裂**：配置（风格/类型/开关/API）在侧栏，运行反馈在主区，注意力需左右跳。

### B. 视觉 / 主题
5. **军事暗色单一、缺层级**：卡片（`result-card`/`log-container`）仅靠 1px 边框区分，无阴影/层次；主标题渐变是唯一视觉锚点，整体偏"扁平工具风"。
6. **设计 token 不完整**：`:root` 只有颜色变量，`render_main` 的 `status_color` 仍用 Python 字符串硬编码；无 spacing/radius/typography 体系，间距（`margin-bottom:16px`/`padding:24px`）散落。
7. **进度条无分段刻度**：`st.progress` 细条，无 5 阶段刻度锚点，与上方阶段灯信息重复但不对齐。

### C. 交互 / 反馈
8. **首启加载无阶段感**：`with st.spinner("…约 45 秒")` 单行文案，无进度/阶段提示（优化 C9 残留）。
9. **结果操作按钮拥挤**：`btn_cols=st.columns(4)`（打开目录/下载/复制/原始版）在窄屏换行混乱。
10. **质检表格不可读**：8 列表格在窄列必溢出，应改用 `st.dataframe`（可横向滚动）或指标卡矩阵。

### D. 可维护性（"不满意却难改"的根因）
11. **1967 行单文件**：UI 渲染 + 流水线编排 + 线程 + subprocess + CSS 全混，`render_*` 强耦合 `session_state`，任何 UI 调整都要通读全文件。这正是"想改 UI 但改不动"的根本原因（对应 AGENTS.md E-3 / WEB_REVIEW D10，Tier 3）。
12. **CSS 与 Python 双轨**：样式一部分在 `_inject_css` 字符串，一部分在 `render_main`/`render_progress` 的 inline `style=`（如 `status_color`），无统一来源。
