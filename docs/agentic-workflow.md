# Agentic Workflow 行为规则

## 模块一：复杂度判定

在开始任何任务前，先做复杂度分级：

| 等级 | 场景 | 工作流 |
|------|------|--------|
| **Tier 1 简单** | 单文件小改、问答、加注释、修 typo | 直接执行，跳过所有协议 |
| **Tier 2 中等** | 多文件修改、新增功能、重构单一模块 | 反射模式（执行→自我审查→修正） |
| **Tier 3 复杂** | 架构设计、系统重构、从零搭建、跨模块协调 | 规划+反射+可选多角色审查 |

**判定方法**：收到任务后，回答以下问题：
1. 涉及文件数 > 3？→ Tier 2+
2. 需要新设计决策？→ Tier 3
3. 影响现有 API/接口？→ Tier 2+
4. 用户明确要求"规划"、"设计"、"重构"？→ Tier 3

如果任务不属于 Tier 2/3，**不要加载本规则的后续模块**，直接执行。

## 模块二：反射协议（Tier 2+ 强制）

执行完成后，按以下 5 维度自我审查：

### 评分体系（每维度 0-20 分，满分 100）

| 维度 | 检查内容 | 0-10 分标志 | 11-15 分标志 | 16-20 分标志 |
|------|----------|------------|-------------|-------------|
| **正确性** | 逻辑正确、边界处理、错误处理 | 有 bug / 漏边界 | 基本正确，少量边缘情况 | 无已知缺陷 |
| **集成性** | 与现有代码风格一致、导入正确、API 兼容 | 风格冲突、破坏接口 | 基本一致，轻微差异 | 无缝集成 |
| **安全性** | 无硬编码密钥、无 SQL/命令注入、输入校验 | 存在安全漏洞 | 基本安全，有改进空间 | 安全最佳实践 |
| **性能** | 时间复杂度合理、无 N+1 查询、无冗余 IO | 明显性能问题 | 可接受，有优化空间 | 性能最优 |
| **可维护性** | 命名清晰、注释适当、函数单一职责、无重复代码 | 混乱难维护 | 基本可维护 | 清晰易读 |

### Reflection 输出格式

每个任务完成后必须输出：

```
[Reflection]
score: <0-100>
missing: <遗漏了什么？如果无则填 N/A>
superfluous: <多做了什么？如果无则填 N/A>
verdict: <PASS | FIXABLE | BLOCKED>
```

### 结论判定

| verdict | 条件 | 行为 |
|---------|------|------|
| **PASS** | score ≥ 80 | 任务完成，无需迭代 |
| **FIXABLE** | score < 80 且问题可自行修复 | 进入迭代修正（模块五） |
| **BLOCKED** | 缺少关键信息、超出能力范围、需用户决策 | 立即停止，列出受阻原因和所需信息 |

**关键规则**：
- 不要为了得到 PASS 而虚高评分
- 发现安全漏洞时，correctness 必须 ≤ 5 分
- BLOCKED 不是失败，是负责任的暂停

## 模块三：规划协议（Tier 3 强制，Tier 2 可选）

### 触发条件
满足以下任一条件时必须先规划再执行：
- 涉及 ≥ 5 个文件的修改
- 新增模块/包/服务
- 架构变更（如引入新模式、更换框架）
- 用户明确要求"先出方案"

### Plan 结构要求

每个 Plan 必须包含：

```
1. 步骤 ID：唯一标识
2. 目标：该步骤要达成的具体结果
3. 涉及文件：具体文件路径列表
4. 依赖：依赖哪些前置步骤
5. 验证标准：如何判断该步骤完成
```

### 输出方式

使用 todo_write 工具输出结构化 TODO 列表，完成后通过 plan_create 或直接向用户呈现计划等待确认。

### 规划审查清单

规划完成后，自问：
- [ ] 所有步骤是否都有明确的依赖关系？
- [ ] 是否存在可并行执行的步骤？（并行 > 串行）
- [ ] 回滚方案是否可行？（不产生不可逆破坏）

## 模块四：多角色审查（Tier 3 可选，按需启用）

仅在 Tier 3 任务且复杂度很高时启用。用不同角色视角审查方案：

| 角色 | 关注点 | 典型问题 |
|------|--------|----------|
| **Code Reviewer** | 代码质量、可读性、命名 | "这段代码半年后还能看懂吗？" |
| **Security Auditor** | 漏洞、认证、授权 | "输入校验是否完备？有无注入风险？" |
| **Architect** | 模块边界、耦合度、扩展性 | "改一个字段会影响多少模块？" |
| **User Advocate** | 用户视角、交互体验 | "用户第一次用能理解吗？" |

**使用协议**：在决策关键节点，以 "作为 [角色名称]..." 开头进行角色切换思考，每个角色不超过 3 条意见。

**注意**：简短的 Tier 2 任务不需要多角色审查，避免过度工程化。

## 模块五：迭代修正（Tier 2+ 当 Reflection 为 FIXABLE 时）

### Evaluator-Optimizer 循环

```
┌─────────────────────────────────────────┐
│                                         │
│  Execution → Reflection(score<80?) → Fix│
│     ↑                                  │
│     └──── 循环: max_iterations=3 ──────┘│
│                                         │
│  任一时刻 score ≥ 80 → 立即退出成功      │
│  达到 max_iterations 仍 < 80 → BLOCKED  │
└─────────────────────────────────────────┘
```

### 迭代规则

1. **max_iterations = 3**：最多修正 3 轮（比默认少，因为 CodeBuddy 上下文成本高）
2. **停滞检测**：连续两轮 improvement < 5 分 → 立即停止，报告 BLOCKED
3. **每轮记录**：记录分数变化、修改内容、改进幅度
4. **每轮 diff report**：总结做了什么修改、为什么这样改

### BLOCKED 报告格式

当迭代终止未达 PASS 时，输出：

```
[BLOCKED Report]
Final Score: <score>
Iterations: <n>
Blockers:
  1. <阻塞原因>
  2. ...
User Action Needed: <用户需要提供什么 / 做什么>
```

## 边界情况说明

| 场景 | 行为 |
|------|------|
| 简单单行修改 | 直接执行，不触发任何协议 |
| 技术问答 / 解释概念 | 直接回答，无需审查 |
| 连续多条简单指令 | 合并为一次 Tier 2 审查 |
| 上下文不足无法判定 | 默认按 Tier 2 处理（宁可多审查，不可漏过） |
| 同一任务跨越多轮对话 | 只在最后一轮执行完整 Reflection |
| 文件超过 500 行需要读全 | 先分段读关键部分 → Reflection 判定是否需要全量 |

## 快速参考卡片

```
Tier 1 → 直接执行
Tier 2 → 执行 + Reflection(5维度评分) + 必要时迭代(max 3轮)
Tier 3 → 规划(Plan) + 执行 + Reflection + 可选多角色审查 + 必要时迭代

PASS      = score ≥ 80 → 完成
FIXABLE   = score < 80 → 迭代修正
BLOCKED   = 无法继续   → 报告受阻原因
```
