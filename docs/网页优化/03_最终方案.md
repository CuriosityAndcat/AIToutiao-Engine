# 最终优化方案（2026-07-12）

> 对应 Agentic Workflow 第三步：审查 Step2 调研方案 → 定稿。
> 审查结论：5 项均属低风险的纯代码修复，集中在 `engine_app.py`，不触碰 `lib/.../backend/` 生产逻辑，不破坏既有解耦。全部采纳，无需裁剪。

---

## 执行清单（Tier 1，直接落地）

### 优化 1 · 进度系统理顺（B4）
- **1a** 在 `_PROGRESS_MAP` 增加 `"download_start": 0.05`。
- **1b** `_download_via_ytdlp` 的 `_progress_hook` 内：
  ```python
  # 旧: st.session_state.progress_pct = 0.05 + p * 0.10
  st.session_state.progress_pct = (
      _PROGRESS_MAP["download_start"]
      + p * (_PROGRESS_MAP["transcribe_start"] - _PROGRESS_MAP["download_start"])
  )
  ```
- **1c** `_PROGRESS_MAP` 键 `research_write_start` → `images_start`（值 0.58 不变）；`step_images` 中引用同步改名。语义：配图阶段起点锚。

### 优化 2 · 响应式增强（C8）
- 在 `_inject_css` 的 `@media (max-width: 768px)` 块补充：窄屏下 stage-indicator 间距收紧、`.result-card` 图片列堆叠友好、`.stat-badge` 允许换行。
- 新增 `@media (max-width: 600px)`：URL 输入行相关组件（`.stTextInput`/按钮）字号与内边距微调，避免极窄屏溢出。

### 优化 3 · 阶段灯文字状态（E13）
- `render_progress` 内新增状态文字映射：`done→完成 / running→进行中 / failed→失败 / pending→待开始`。
- 指示灯 HTML 在名称下方追加状态文字（小号、对应颜色），强化非颜色通道的辨识。

### 优化 4 · 进度百分比（交互）
- `render_progress` 在 `st.progress(...)` 后追加 `st.caption(f"进度 {int(progress_pct*100)}%")`，精确反馈。

### 优化 5 · 异常分层注释（D12）
- `main()` 线程 `_run` 的 `except BaseException` 加注释：必须捕获 BaseException 以兜底线程级致命错误并**确保 `_PIPELINE_LOCK` 释放**。
- `execute_pipeline` 外层 `except Exception` 加注释分层说明（业务异常 vs 致命异常）。

## 验收锚点（供 Step4 使用）
- 进度条在「下载(yt-dlp)」阶段从 5% 平滑到达转录起点，无内联魔法数字。
- 阶段指示灯显示「完成/进行中/失败/待开始」文字。
- 进度条下方显示百分比。
- 窄屏（≤768px / ≤600px）布局不溢出、可堆叠。
- 代码无新增外部依赖，改动仅限 `engine_app.py`。
