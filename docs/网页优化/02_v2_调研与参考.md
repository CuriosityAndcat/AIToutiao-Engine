## 一、可复用的标准 / 官方范式

### 1.1 Streamlit 官方布局与容器（核心依据）
- **官方文档《Layouts and Containers》**：明确推荐用 `st.tabs` / `st.columns` / `st.expander` / `st.container` 组织界面，而非单页长滚动。
- **`st.tabs`**：天然解决 Step 1 问题 A1（单页无分区）——把「运行监控 / 成果展示 / 配置」分区。
- **`st.container(height=...)`**：成果稿件已用（`height=500`），可继续用于日志/稿件受限滚动区。
- **`st.status`**（Streamlit 1.3+）：适合替代首启 `st.spinner` 单行文案，提供「加载中→完成」分阶段状态（解决 C8）。

### 1.2 设计 Token 体系
- Streamlit 官方支持 `.streamlit/config.toml` 主题变量，但本项目已用 `_inject_css` 的 `:root` 自定义变量（WEB_REVIEW D11 已做颜色 token 化）。
- **补强方向**：在 `:root` 增加 spacing / radius / typography token，消除 `render_main`/`render_progress` 中的 inline `style=` 硬编码（解决 B6/B12）。
- 参考：Microsoft `Streamlit_UI_Template`（WEB_REVIEW §6.1）的 CSS 变量分层思路（仅参考设计，不复制代码，避免与军事风冲突）。

### 1.3 增强组件库（可选，非必需）
- **`streamlit-extras`**（社区库，PyPI 持续维护）：提供 `st_keyup`、`card`、`switch_page` 等增强组件。
- **取舍**：本项目重设计**优先用原生组件**（`st.tabs`/`st.dataframe`/`st.status`），不强制引入 `streamlit-extras`，保持零新增硬依赖（对齐 WEB_REVIEW「不引入外部库」约束）。仅当原生方案明显不足时，把 `streamlit-extras` 列为可选依赖。

## 三、方案方向裁定（收敛 Step 1 预判）

| Step 1 预判方向 | 是否有标准范式 | 裁定 |
|------------------|----------------|------|
| 顶部 Tab 分区 | ✅ 官方 `st.tabs` | 采用，零依赖 |
| 卡片化 / 阴影层级 | ⚠️ 仅 CSS（参考 MS 模板） | 采用纯 CSS，不引库 |
| 设计 token 补全 | ✅ 自有 `:root` 扩展 | 采用，消除 inline 硬编码 |
| 质检表改 `st.dataframe` | ✅ 官方组件 | 采用，解决窄列溢出 |
| 运行态闪烁优化 | ⚠️ fragment 机制本身，需调 `key`/范围 | 采用局部刷新 |
| 首启阶段感 | ✅ 官方 `st.status` | 采用 |
| 单文件拆分 | ⚠️ code-splitter（Tier 3） | **单列 Track 4，不在本轮执行** |

**结论**：本重设计**完全可基于 Streamlit 原生能力 + 自有 CSS token 完成**，无需引入新依赖；仅 Track 4（拆分）属 Tier 3，需独立立项。

## 四、风险与约束

1. **不触碰生产逻辑**：仅改 `engine_app.py` 的 UI 层（`render_*` + `_inject_css` + `main` 分支），不碰 `lib/.../backend/`。
2. **运行态 fragment 兼容**：`st.tabs` 与 `@st.fragment(run_every=3)` 需共存——fragment 应落在「运行监控」tab 内，避免切 tab 丢刷新。
3. **零新增硬依赖**：`streamlit-extras` 仅作可选备注，不写进 `requirements.txt`。
4. **军事暗色主题保留**：视觉升级在现有军事风框架内做"层级与分区"，不换成浅色/其他主题（避免与品牌冲突，除非用户要求）。
